# Гайд — Навички Meweek

Це список того, що вже можна просити в агентів у Meweek. Тут тільки людське пояснення: для чого навичка, коли її кликати, як сформулювати запит і що має з'явитися в результаті.

Якщо навички або команди немає тут, вважай, що вона ще не готова для використання. Службові правила для агентів живуть у [[Конектори і навички]].

## Швидкий список

| Назва | Як просити | Для чого | Статус |
|---|---|---|---|
| `git-sync` | `/sync` або “синхронізуй” | синхронізувати vault через git | працює |
| `save-context` | `/save-context` або “збережи контекст” | зберегти коротку пам'ять проєкту перед паузою | працює |
| `new-project` | `/new-project <назва>` | завести новий проєкт у `10-Projects/` | працює |
| `weekly-review` | `/weekly-review` | навести лад у сховищі раз на тиждень | працює |
| `docker-project` | “зроби docker для <проєкту>” | створити Docker-оточення | працює |
| `model-budget` | `/model-budget <задача>` | вибрати модель Claude під задачу | працює |
| `skill-forge` | `/skill-forge впровадь <ідея>` | перетворити ідею на готову навичку | працює |
| `skill-creator` | через `skill-forge` | допоміжний інструмент для створення навичок | технічний інструмент |
| `fitness` | `/fitness Микола` або `fitness Микола` | вести Fitness-проєкт | працює |
| `billing-monitor` | поки не запускати | моніторинг витрат Anthropic | ідея, ще не працює |

## Щоденна робота

### `/sync` — синхронізація

**Для чого.** Щоб локальні файли й GitHub були в одному стані.

**Коли просити.** На початку роботи, наприкінці роботи або коли треба запушити коміти.

**Як сказати.**

```text
/sync
```

або:

```text
синхронізуй середовище
```

**Що буде результатом.** Агент перевірить `git status`, підтягне зміни, закомітить або запушить те, що треба. Якщо є конфлікти чи чужі незакомічені файли, має зупинитись і сказати.

### `/save-context` — зберегти пам'ять проєкту

**Для чого.** Щоб після `/clear` або наступної сесії агент не починав з нуля.

**Коли просити.** Перед паузою, завершенням роботи або зміною теми.

**Як сказати.**

```text
/save-context
```

або:

```text
збережи контекст Fitness
```

**Що буде результатом.** Оновлений файл `<Проєкт> — Контекст.md`: стан, останні рішення, наступний крок і відкриті питання.

## Проєкти

### `/new-project` — завести новий проєкт

**Для чого.** Щоб ідея стала нормальним проєктом з папкою, оглядом, планом і контекстом.

**Коли просити.** Коли з'явилась нова тема, яку треба вести окремо.

**Як сказати.**

```text
/new-project Назва
```

або:

```text
створи проєкт Назва
```

**Що буде результатом.** Папка в `10-Projects/<Назва>/`, базові файли проєкту, запис у списку проєктів і коміт.

### `docker-project` — Docker для проєкту

**Для чого.** Щоб проєкт запускався локально через Docker: база, веб, кеш, адмінка або інші сервіси.

**Коли просити.** Коли проєкту потрібне стабільне локальне оточення.

**Як сказати.**

```text
зроби docker для DBManager
```

або:

```text
підніми оточення для <проєкту>
```

**Що буде результатом.** `docker-compose.yml`, `.env.example`, `.dockerignore`, за потреби `Dockerfile`, команди запуску в огляді проєкту. Реальні секрети не мають потрапити в git.

### `/fitness` / `fitness` — Fitness-проєкт

**Для чого.** Вести щоденники Fitness для Миколи й Марти: вага, їжа, тренування, сон, алкоголь, звички, тижневі підсумки.

**Коли просити.** Коли є щоденний запис, уточнення або треба підбити тиждень.

**Як сказати.**

```text
fitness Микола вага 81.4, тренування пропустив, їв ...
```

```text
fitness Микола підбий тиждень
```

**Що буде результатом.** Оновлений журнал і профільні файли Fitness. Для “що робити зараз” відкривай [[Микола — План дій]] або [[Марта — План дій]].

## Обслуговування сховища

### `/weekly-review` — тижневий розбір

**Для чого.** Навести лад у vault: inbox, контексти, биті посилання, застарілі файли.

**Коли просити.** Раз на тиждень або коли стало важко знайти актуальний стан.

**Як сказати.**

```text
/weekly-review
```

або:

```text
зроби тижневий огляд сховища
```

**Що буде результатом.** Розібрані нотатки, оновлені контексти або список того, що треба вирішити вручну.

### `/review` — перевірка змін

**Для чого.** Отримати рев'ю перед комітом: помилки, ризики, пропущені тести, небезпечні зміни.

**Коли просити.** Перед важливим комітом, особливо для DBManager, безпеки або продакшн-коду.

**Як сказати.**

```text
/review
```

або:

```text
перевір ці зміни перед комітом
```

**Що буде результатом.** Список знахідок за важливістю. Якщо проблем немає, агент має прямо сказати, що критичних знахідок не бачить.

## Моделі й витрати

### `/model-budget` — вибір моделі

**Для чого.** Зрозуміти, яку модель Claude брати під задачу: дорожчу й сильнішу чи дешевшу для рутини.

**Коли просити.** Перед створенням рутини, масовою обробкою, дорогим запуском або коли є сумнів “якою моделлю це робити”.

**Як сказати.**

```text
/model-budget щоденний дайджест новин
```

або:

```text
яку модель взяти для рев'ю DBManager?
```

**Що буде результатом.** Рекомендована модель, коротке пояснення й орієнтир вартості. Якщо запустити без аргументів, агент має показати коротку таблицю вибору моделей.

## Розвиток навичок

### `/skill-forge` — зробити або покращити навичку

**Для чого.** Перетворити ідею з `30-Agents/Skill Ideas/` на готову навичку або покращити вже наявну.

**Коли просити.** Коли є повторювана дія, яку не хочеться щоразу пояснювати заново.

**Як сказати.**

```text
/skill-forge впровадь billing-monitor
```

або:

```text
/skill-forge покращ model-budget
```

**Що буде результатом.** Якщо ідея варта реалізації: службовий `SKILL.md`, команда за потреби, запис у цьому гайді, оновлений реєстр агентів і коміт. Якщо ідея ризикована або ще сира, агент має сказати чому й не робити вигляд, що вона готова.

### `skill-creator` — технічний інструмент для навичок

**Для чого.** Це не повсякденна команда, а інструмент Anthropic для створення й перевірки якості навичок.

**Коли просити.** Зазвичай не треба напряму. Проси `/skill-forge`, а він уже використає `skill-creator`, якщо потрібно.

**Як сказати.**

```text
створи навичку через skill-forge
```

**Що буде результатом.** Не окремий користувацький результат, а якісніше оформлена навичка всередині `30-Agents` / `.claude/skills`.

## Ідеї, які ще не працюють

### `billing-monitor`

**Що це мало б робити.** Показувати стан використання Anthropic-кредитів і попереджати, якщо витрати наближаються до ліміту.

**Поточний стан.** Це лише ідея в `30-Agents/Skill Ideas/billing-monitor.md`, не готова команда і не активна навичка.

**Чому ще не працює.** Потрібно вирішити, звідки брати billing-дані: API, web UI або ручний ввід. Без цього агент не має надійного джерела правди.

**Як запустити розробку.**

```text
/skill-forge впровадь billing-monitor
```

Після впровадження тут має з'явитися нормальний опис, команда запуску й очікуваний результат.

## Як перевірити, що гайд не забули

Після появи нової навички або команди має бути:

1. короткий запис у цьому файлі;
2. команда в [[Гайд — Команди Meweek]], якщо її можна запускати вручну;
3. службовий запис у [[Конектори і навички]];
4. коміт із цими змінами.

Пов'язане: [[Гайди]], [[Гайд — Команди Meweek]], [[Конектори і навички]].
