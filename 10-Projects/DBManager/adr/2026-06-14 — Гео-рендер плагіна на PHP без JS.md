# ADR: Гео-рендер у плагіні DBManager на PHP (без JS), MaxMind через bridge

Проєкт: [[DBManager — Огляд]] · Дизайн: [[DBManager — Дизайн]] (§8, §10) · План: `plans/2026-06-14 — План 1.3 — WP-плагін.md`

## Контекст

Дизайн §8/§10 для гео-залежних значень обрав **рендер через JS** (нейтральний span-плейсхолдер → AJAX до свого ж сайту → визначення країни → підстановка), бо на більшості сайтів є повносторінковий кеш: серверний PHP на закешованій сторінці не виконується, тож усі відвідувачі бачили б однакове значення.

Користувач ухвалив протилежне обмеження: **плагін виводить дані лише через PHP і власні функції, без жодного JS** — включно з гео-детектом. Це підсилює пріоритет №1 дизайну (анти-fingerprint: у публічному HTML нуль маркерів системи, нічого виконуваного на клієнті).

## Розглянуті варіанти

1. **JS/AJAX (як у дизайні §8)** — працює поверх будь-якого page-cache без його перенастройки; мінус — у HTML є клієнтський скрипт (слабший анти-fingerprint), залежність від JS на клієнті.
2. **PHP geo-detect (обрано)** — країна з заголовка `CF-IPCountry` (Cloudflare), фолбек — локальний лукап по MaxMind-базі, далі «Світ»; рендер значень у PHP за гео-видимістю. Мінус — щоб гео-значення різнились по країнах при повносторінковому кеші, **кеш має враховувати країну** (CF-IPCountry у ключі кешу) або гео-сторінки виключаються з кешу.

## Рішення

Варіант 2, **повне гео в плані 1.3**, без JS:

- **Гео = видимість.** Контракт payload (компілятор Core): кожне значення несе `geo` (коди країн або `WORLD`). Значення показується, якщо детектована країна ∈ `geo` або `geo` містить `WORLD`; інакше рендериться порожнім. Логіка — у спільному ядрі `dbm_render_from_cache` (PHP, без JS).
- **Джерело країни (PHP):** `CF-IPCountry` → локальний MaxMind-лукап IP → `WORLD`.
- **MaxMind-база роздається центром через bridge** (дзеркалить пайплайн даних): Core публікує `.mmdb` у bridge (HMAC), bridge віддає її плагіну по токену (ETag за хешем + підпис), плагін завантажує, верифікує, кешує локально (`wp-content/uploads/...`, переживає видалення плагіна) і робить лукап через **вбудований чистий-PHP `maxmind-db/reader`** (self-hosted, нуль зовнішніх runtime-викликів).

## Наслідки

- **Плюс:** сильніший анти-fingerprint — у публічному HTML немає скриптів/маркерів; нуль зовнішніх ресурсів; рендер повністю серверний.
- **Вимога до деплою (ризик):** повносторінковий кеш має варіюватися за `CF-IPCountry` (ключ кешу) **або** гео-сторінки виключаються з кешу — інакше відвідувачі бачитимуть країну першого, хто закешував сторінку. Це питання конфігурації CDN/кешу (DevOps), не коду плагіна. Не-гео значення (`WORLD`) кешем не зачеплені.
- **Обсяг 1.3 зростає:** додаються таски на стороні Core (публікація `.mmdb`), bridge (geodb-ендпоінт), плагіна (синхронізація бази + детектор + гео-видимість рендеру). Деталі — у плані 1.3.
- Інтерфейс детектора країни лишається швом: якщо колись повернути JS-варіант для окремих кейсів, його можна додати поряд, не ламаючи PHP-рендер.
