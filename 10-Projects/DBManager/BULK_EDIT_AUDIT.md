# BULK_EDIT_AUDIT — критичний аудит масових змін

> Мета: зробити масові зміни **передбачуваними і майже без плутанини**. Посилання — `file:line` від `code/core/`.
> Пов'язано: [[DATA_TYPES_MAP]], [[NEW_TYPES_PLAN]], [[DBManager — Контекст]].

## 1. Як це працює зараз (потік)

1. **Вибір сайтів** — scope `all/group/tree/selected`; `editableTargetSites()` фільтрує по `canEditSite && canPublishSite` (`BulkOperations.php:782-788`).
2. **Фільтр цілей** — `targetType` + `state/geo/search/phone` (`matchedDataValues:449`, `matchedNumberEntries:470`).
3. **Операція** — 7 операцій, диспетч **за операцією, не за типом** (`isPhoneOperation:815`): VALUE-конвеєр (`replace_text/set_value/set_status/set_geo/set_phone_format`) і PHONE-конвеєр (`replace_phone/set_phone_status`).
4. **Прев'ю** — `previewRows()` будує **перші 80** рядків (`->take(80)`, `554-631`) з diff `state/geo/key/value/format`.
5. **Підтвердження + застосування** — `wire:confirm` (статичний рядок), `apply()` мутує в транзакції, пише 1 `AuditLog` на запис зі спільним `batch_id`, потім **синхронно публікує** (`publishAfterApply=true` за замовч., `64`).
6. **Звіт + відкат** — останні 3 батчі (`getRecentBulkSessions`), `rollbackBatch → AuditRestorer::restore`.

## 2. Що працює добре

- **Diff-підсвічування в прев'ю** — реальне попарне «старе → нове» по клітинках (strike-червоне старе, зелене нове), see `bulk-operations.blade.php:269-363`. Це найсильніший UX-елемент.
- **Атомарність + аудит** — мутація в транзакції, кожна зміна логується, є відкат через відтворення `batch_id` у зворотному порядку.
- **Generic-таргетинг** — новий тип фільтрується коректно без коду (`matchedDataValues` через `whereHas('type','code')`).

## 3. Де користувач помиляється / не розуміє (критичні точки)

| # | Де | Проблема | Тяж. |
|---|---|---|---|
| C1 | `targetType` `<select>` (`blade:155-162`) | text/address/social **пропонуються**, але `set_value` завжди пише `content['value']` (`247`), а їхня реальна форма інша → тиха псування/no-op без попередження | 🔴 |
| C2 | Прев'ю `take(80)` vs apply на **весь** набір | користувач рев'юить 80 рядків і думає, що це весь набір змін; банер «обмежено» легко пропустити, а `wire:confirm` не повторює кількість | 🔴 |
| C3 | `replace_text` (`replaceRecursively:820-842` + ключ `237`) | **нескоупований** find/replace по ключу І всіх вкладених рядках `content`; коротке `find` («380», «ua») тихо переписує формати, url, гео-коди | 🔴 |
| C4 | `wire:confirm` (`blade:503`) | один статичний рядок без операції, типу, кількості, find/replace — нічого для перевірки перед незворотною дією | 🔴 |
| C5 | `set_value` завжди `changed=true` (`245-249`) | прев'ю показує «зміну» навіть коли значення ідентичне → зайві audit-рядки + republish | 🟡 |
| C6 | `set_phone_status` на `phone_reserve` | без обов'язкового phone-фільтра можна одним кліком змінити статус **усіх** резервів на сайтах | 🟡 |
| C7 | `publishAfterApply` за замовч. ON | зміни йдуть **наживо** на сайт миттєво при apply; користувач, що грається з фільтрами, може не усвідомлювати | 🟡 |
| C8 | Відкат лише для 3 останніх батчів (`934-942`) | після 4-ї операції найперший відкат зникає з UI | 🟡 |

## 4. Ризики неоднозначності між типами

- `displayValue` ELSE-гілка (`870`) згортає messenger/text/social/address у `content['value']??['url']` — соц-url і текст **невідрізнянні** в прев'ю, тип з іншою формою показує `'—'`.
- `set_value` пише `content['value']` незалежно від типу (`247`) — для url-типів створює мертве поле, а реальне лишається; audit `new` розходиться з тим, що читає публікація.
- `set_geo` має **дві** моделі гео в одній операції: pivot `geoTags` (default) vs `content['prices'][].geo` (price) — новий тип із гео в `content` тихо не зміниться.
- Псевдотип `phone_reserve` — alias на `phone` (`461`) + split по `NumberEntry.priority` (`484-489`); його немає у `value_types` — пастка для читача.
- У прев'ю `row['type']` для телефонів береться з `entry.priority` (`582`), а для значень — з `value.type.code`: **два джерела** одного поля.
- Фільтр стану `down` — phone-only (`516-523`), але пропонується для всіх типів → фільтр не-phone по «Збій» тихо дає 0 без пояснення.

## 5. Структурні зауваги

- **`BulkOperations` НЕ використовує** `HandlesScopeDecision`/`UsesEditLock` (підтверджено grep) — bulk обходить scope-діалог і edit-lock, які мають single-edit потоки → слабший захист від конкурентності/scope, ніж в одиночному редагуванні.
- Немає поділу dry-run / commit: apply одразу мутує **і** публікує синхронно в одному реквесті; великий набір блокує UI.
- `set_phone_status` rollback резолвить entry по **глобальному** e164 (`AuditRestorer.php:373-388`) — неоднозначно для спільних номерів.

## 6. Пропозиції з пріоритетами

### P0 — прибрати тиху псування і непередбачуваність
- **B1. Гейт типів/операцій (або реєстр хендлерів).** Прибрати з `targetType` (`blade:155-162`) типи, для яких operation тихо ламає, **поки** вони не підключені; або ввести інтерфейс `ValueTypeBulkHandler` (resolveDisplay / applySetValue / applyReplace / geoStorage), щоб тип декларував форму `content` один раз і однаково проходив apply+preview+rollback. *(Закриває C1.)*
- **B2. Багате підтвердження.** Замінити `wire:confirm` на модалку-резюме: операція, тип, **точна** кількість цілей, find/replace-рядки, явна примітка «переглянуто 80 з N». *(Закриває C2, C4.)*
- **B3. Скоупити `replace_text`.** Селектор «де шукати» (лише ключ / лише `content.value` / усе) + показ у прев'ю **якого саме поля** торкнулась заміна. *(Закриває C3.)*

### P1 — точність і безпека
- **B4. Guard проти no-op.** `set_value`/`set_geo` позначати `changed` лише при реальній відмінності (як `set_status:250-254`). *(C5.)*
- **B5. Відокремити apply від publish.** `publishAfterApply` за замовч. OFF або окремий крок «Опублікувати зараз». *(C7.)*
- **B6. Повна історія відкату + надійний phone-rollback.** Показувати всі батчі (не 3); резолвити rollback по збереженому `entry_id/data_value_id`, не по рядку номера. *(C6, C8.)*

### P2 — масштаб
- **B7. Черга + спільні захисти.** Винести apply у job із чанкінгом і прогресом, провести через ті ж scope-decision/edit-lock, що й single edit.

## 7. Підсвічування змін — поточний стан і прогалини

**Реалізовано (лише прев'ю, `bulk-operations.blade.php`):** state (`269-277`), geo (`281-312`, додані коди `bg-ok-bg`, негативні `!` `bg-bad-bg`), key (`315-323`), value (`326-341`, `$showChangedLayout = changed && value!==new_value`), format (phone, `344-363`). `new_*` рахуються в `calculateNewValue/calculateNewPhone:633-765`.

**Прогалини (вимога користувача — підсвічувати ЗАВЖДИ):**
1. Підсвічування **лише в прев'ю** — звіт (`520-542`) і список «Останні зміни» (`579-585`) показують plain-текст.
2. Лише перші 80 рядків мають diff; рядки 81..N мутуються без візуалізації.
3. Для `replace_text` diff на рівні агрегату — не показує, яке вкладене поле змінилось.
4. **Single edit — нуль підсвічування** (див. [[DATA_TYPES_MAP]] §3.7): редактор не показує «старе → нове».
5. address/social custom-поля невидимі для diff (`displayValue` дає `'—'`).
6. Колір — єдиний сигнал (без «було/стало»), легка проблема доступності.

**Рекомендація щодо §4 завдання:** ввести єдиний компонент diff-клітинки (одне джерело істини «ця клітинка змінилась»), застосувати його і в bulk-прев'ю, і в single edit (показ оригіналу), і зробити `displayValue` змістовним для нових типів — без цього нові типи дають оманливе `— ▼ —`.
