# Архітектура DBManager — і чому три URL

Коротка відповідь на питання «чому в dev три URL, якщо ми планували CRM і плагін у WP». Третій URL — це **DataBridge**, навмисний ізоляційний шар між центром і сайтами. Він не зайвина, а ключ до безпеки всієї системи.

Джерела: [[DBManager — Огляд]] · [[DBManager — Дизайн]] (§1 іменування, §3 архітектура, §5 доставка, §8 плагін, §9 безпека).

## 1. Три компоненти

| Компонент                                         | Роль                                                                                                                                          | Де живе                                                        | Dev-URL                                                                                            |
| ------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| **DBManager Core** (Laravel)                      | Центр: адмінка (CRM) + БД + failover-движок + аудит/інциденти. Тут редагуються дані.                                                          | Внутрішня зона (у проді — VPN/allowlist, з інтернету невидима) | http://localhost:8001 (вхід `admin@dbmanager.local` / `admin`)                                     |
| **DataBridge** (Laravel)                          | Публічний ізоляційний шар. Тримає read-only **копію** опублікованих даних. Сайти читають дані саме з нього по токену.                         | Публічна зона (доступна з інтернету)                           | http://localhost:8002 — це **API, не сайт**: корінь віддає 404, здоров'я на `/up` та `/api/v1/...` |
| **WP-плагін «DBManager»** (код: DBManager Client) | Read-only клієнт у кожному сайті: тягне дані з DataBridge по токену, кешує у `wp_options`, рендерить суто PHP (без JS), має mu-plugin фолбек. | На самих сайтах WordPress (у проді — 50+ доменів)              | Зразковий WP для тесту: http://localhost:8080 (вхід WP `admin`/`admin` → меню «DBManager»)         |

Важливо: `:8080` — це **не наш окремий сервер і не четвертий продукт**. Це один зразковий WordPress, піднятий у dev як стенд, щоб було де перевірити плагін. У проді плагіна стоятиме на багатьох реальних сайтах, кожен зі своїм доменом.

## 2. Чому три, а не два

Продукт = **Core (CRM)** + **DataBridge (ізоляція)** + **плагін (на сайтах)**. DataBridge існує заради безпеки, а не для зручності.

**Односторонній потік знань** (§3 дизайну). Плагін на сайті знає лише дві речі: URL DataBridge і свій токен. Він **нічого не знає про Core**. Дані рухаються лише в один бік: Core *пушить* опубліковану копію в Bridge (з HMAC-підписом і версією), а Bridge **не має доступу до БД Core** — у нього окрема власна база (`bridge-mysql`).

Наслідок для безпеки: **злам публічного Bridge не розкриває внутрішній Core.** Атакувальник, який дістався до Bridge або до коду плагіна на чужому сайті, бачить лише вже опубліковані, «порожні» дані одного сайта — жодного сліду центральної системи, жодного доступу до редагування чи до решти сайтів. Це прямо обслуговує наскрізний пріоритет №1: по плагіну/HTML неможливо відстежити центр.

Якби сайти ходили в Core напряму (два URL), центр довелося б відкрити в інтернет — і кожен сайт став би потенційним містком до нього. Bridge розриває цей місток.

## 3. Потік даних

```
   Зовнішній моніторинг
   «номер впав» (webhook,
    окремий секрет)
          │
          ▼
┌──────────────────────┐   HMAC push    ┌──────────────────────┐
│  DBManager Core       │  (версія,      │  DataBridge           │
│  (внутрішня зона)     │   payload      │  (публічна зона)      │
│                       │   ураженого    │                       │
│  адмін редагує/       │   сайта)       │  read-only копія      │
│  failover-движок ───► │ ─────────────► │  опублікованих даних  │
│  компілює payload     │                │  rate limit · черга   │
└──────────────────────┘                └──────────┬───────────┘
                                                    │
                          push-пінг «оновись» ──────┤
                          (лише ураженим сайтам)    │
                                                    ▼
                                         ┌──────────────────────┐
                                         │  WP-плагін DBManager  │
                                         │  на сайтах            │
                                         │                       │
                          дані по токену │  перевіряє підпис +   │
                          ◄───────────── │  ETag, рендерить      │
                                         │  з кешу (wp_options)  │
                                         └──────────────────────┘
                          + страхувальна звірка 1/добу
```

Кроки (§5 дизайну):
1. Адмін у Core змінює значення **або** приходить webhook «номер впав» → failover-движок перемикає слот.
2. Core компілює готовий payload **уражених** сайтів і пушить у Bridge (HMAC, монотонна версія).
3. Bridge шле push-пінг лише ураженим сайтам.
4. Плагін одразу тягне свіже з Bridge (перевірка підпису + `If-None-Match`/ETag) і рендерить з кешу.
5. Плюс добова звірка «моя версія актуальна?» — ловить загублені пінги і слугує пульсом парку.

Постійного опитування немає; ціль failover — **≤ 1 хв**.

## 4. Dev проти прод

| | Dev (локальна машина) | Прод |
|---|---|---|
| **Core** | http://localhost:8001, креди `admin`/`admin` | Внутрішня зона: VPN/allowlist, з інтернету невидима; справжні креди |
| **DataBridge** | http://localhost:8002 (API) | Публічний хост; з ростом виноситься на окремий сервер без зміни коду |
| **Плагін** | один зразковий WP на http://localhost:8080 | На 50+ реальних сайтах, кожен зі своїм доменом і токеном |

Безпека dev-середовища: усі порти в `docker-compose` прив'язані до `127.0.0.1` (доступні лише з локальної машини); креди `admin`/`admin` — суто для локальної розробки і в проді змінюються; секрети живуть у `.env` (поза git).

## 5. Запуск і тести

Усе підіймається через Docker Compose (сервіси `core`, `mysql`, `bridge`, `bridge-mysql`, `redis`, `wordpress`, `wpcli`, `plugin`). Команди — з каталогу `C:/Dev/Meweek/10-Projects/DBManager/code`:

```
docker compose run --rm core   php artisan test
docker compose run --rm bridge php artisan test
docker compose run --rm plugin ./vendor/bin/phpunit
```
