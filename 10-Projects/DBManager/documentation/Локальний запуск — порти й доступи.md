# Локальний запуск — порти й доступи

Шпаргалка для локальної (dev) розробки DBManager: які сервіси де слухають, як зайти, як запустити й прогнати тести. Контекст системи — [[DBManager — Огляд]]; джерело правди по архітектурі — [[DBManager — Дизайн]].

Чому портів три (а не два): продукт = **DBManager Core** (CRM/адмінка) + **DataBridge** (публічний ізоляційний шар) + **WP-плагін** на сайтах. Bridge існує заради безпеки — «односторонній потік знань»: плагін знає лише URL Bridge + токен, а злам публічного Bridge не розкриває внутрішній Core (§3, §9 дизайну). У dev ми додатково тримаємо **один зразковий WordPress** (`:8080`) як стенд для плагіна — це «де живе плагін для тесту», а не четвертий продукт. У проді плагін стоятиме на 50+ реальних сайтах, кожен зі своїм доменом.

## Порти й доступи

| Сервіс | URL / порт | Що це | Логін |
|---|---|---|---|
| CRM (DBManager Core) | http://localhost:8001 | Адмінка: значення, сайти, failover, аудит, інциденти. Внутрішня зона (у проді — VPN/allowlist) | `admin@dbmanager.local` / `admin` |
| DataBridge | http://localhost:8002/up | Публічний read-only API (не UI). Корінь `/` віддає 404 — це норма; здоров'я на `/up` та `/api/v1/...` | без UI-логіну (доступ по токену сайта) |
| WordPress-стенд | http://localhost:8080/wp-admin | Зразковий сайт із встановленим плагіном «DBManager» (меню «DBManager») | `admin` / `admin` |
| MySQL (Core) | 127.0.0.1:33061 | БД Core; тут же база `dbmanager_wp` зразкового WordPress-стенда (окремого MySQL для WP немає) | з `.env` |
| MySQL (Bridge) | 127.0.0.1:33062 | Окрема БД Bridge (ізоляція: Bridge не торкається БД Core) | з `.env` |
| Redis | без публікації порту | Службова черга/кеш, доступний лише в мережі Compose | — |

> Порти `8001`/`8002` — це **host-порти**; усередині контейнерів Core/Bridge застосунок слухає `:8000` (мапінг `127.0.0.1:8001:8000` тощо). З браузера звертайся саме до host-портів 8001/8002/8080.

Адмінка Core у роботі (план 2.1: поки вхід + оболонка; грід значень — наступні таски). DataBridge і WP-плагін — готові (плани 1.2 і 1.3).

## Як підняти

Усі команди — з каталогу коду:

```bash
cd C:/Dev/Meweek/10-Projects/DBManager/code
docker compose up -d
```

Сервіси Compose: `core`, `mysql`, `bridge`, `bridge-mysql`, `redis`, `wordpress`, `wpcli`, `plugin`.

Тести:

```bash
# Core
docker compose run --rm core php artisan test

# DataBridge
docker compose run --rm bridge php artisan test

# WP-плагін
docker compose run --rm plugin ./vendor/bin/phpunit
```

## Безпека dev

- Порти в `docker-compose` прив'язані до `127.0.0.1` — сервіси видно лише з локальної машини, не з мережі. Щоб зміна прив'язки застосувалась: `docker compose up -d`.
- Креди `admin/admin` (Core та WordPress-стенд) — **лише для локального dev**. У проді змінюються.
- Секрети — у `.env` (поза git); у репо лише `.env.example` без значень.

## Типова помилка

**`Connection refused 127.0.0.1:3306`** — найчастіша пастка.

- Заходь у застосунок через **контейнер** (порти `8001` / `8002`), а не через локальний `php artisan serve` на хост-машині. Локальний PHP не бачить БД у мережі Compose.
- `DB_HOST` у `.env` має бути **іменем сервісу** (`mysql` для Core, `bridge-mysql` для Bridge), а не `127.0.0.1`. Laravel читає змінні лише з `.env`, а не з OS-env, які виставляє Compose, — тому host треба прописати у `.env` явно.
