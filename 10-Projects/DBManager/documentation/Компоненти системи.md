# Компоненти системи

Система DBManager складається з **трьох** компонентів, а не двох. Це часта точка плутанини: окрім центру і плагіна є ще проміжний публічний шар **DataBridge** — той самий «третій URL», який існує заради безпеки й ізоляції. Деталі архітектури — [[DBManager — Дизайн]] (§3); загальний контекст — [[DBManager — Огляд]].

Коротко про потік: у Core адмін редагує значення або приходить webhook «номер впав» → Core компілює payload ураженого сайта і **пушить** його в DataBridge (HMAC, версія) → Bridge шле push-пінг сайту → плагін одразу тягне свіже з Bridge (перевірка підпису + ETag) → рендерить з кешу. Плюс добова звірка. Ціль failover — **≤ 1 хв**. Постійного опитування немає.

---

## 1. DBManager Core

**Роль.** Центр системи: адмінка (CRM) для відділу + база даних + failover-движок + аудит та інциденти. Тут редагуються всі дані; назовні нічого не віддає напряму — лише пушить опубліковані дані в DataBridge.

**Де живе.** Внутрішня зона. У проді невидимий з інтернету (VPN/allowlist). Dev-доступ: http://localhost:8001 (вхід `admin@dbmanager.local` / `admin`).

**Стек.** PHP / Laravel, MySQL, Redis. Запуск у Docker Compose.

**Що вже зроблено (плани 1.1, частина 2.1):**
- Модель даних: значення з гео-мітками, групи й перекриття сайтом, телефонні слоти з ланцюгом резервів.
- Failover-движок (`SlotResolver` + `FailoverEngine`): падіння/відновлення номерів, sticky, ручний pin, спільні номери без дублів, дедуплікація інцидентів.
- Компілятор payload (`SitePayloadCompiler`): готові дані сайта з урахуванням override «сайт > група», месенджерів, прихованих значень, версіонування.
- Webhook моніторингу «номер неактивний» (HMAC, throttle проти failover-шторму, транзакційність).
- Токени сайтів (`SiteProvisioner`: read-only токен на свої дані, ping_url).
- Публікатор у DataBridge (`BridgePublisher`): пуш опублікованих даних після failover/зміни.
- Адмінка щойно почалась (план 2.1: поки оболонка; повний UI, ролі, імпорт CSV/XLSX, аудит з відкатом — Етап 2).

**Безпека.** Мережева ізоляція адмінки; HMAC на webhook; токени read-only з відкликанням і ротацією; аудит дій; секрети в `.env` поза git. У dev порти прив'язані до `127.0.0.1`, креди `admin/admin` — лише локально, у проді змінюються.

## 2. DataBridge

**Роль.** Публічний ізоляційний шар. Тримає **read-only копію** опублікованих даних. Сайти читають дані саме з нього — по токену. Core пушить дані в Bridge (HMAC); сам Bridge **не має доступу до БД Core**. Це і є «третій URL»: публічна точка, яку очікують найменше, але саме вона забезпечує безпеку.

**Де живе.** Публічна зона. Це **API, не сайт**: корінь відповідає 404, здоров'я — на `/api/v1/...` та `/up`. Dev: http://localhost:8002.

**Стек.** Мінімальний Laravel + власна окрема `bridge-mysql` (фізична ізоляція від БД Core). Запуск у Docker Compose.

**Що вже зроблено (план 1.2):**
- Ingest Core → Bridge по HMAC з монотонністю версій (атомарний `lockForUpdate`).
- Видача даних по токену з HMAC-підписом + ETag.
- Звірка `If-None-Match` → 304 (легка перевірка «моя версія актуальна?», ~200 байт).
- Push-пінги сайтам із HMAC і наростаючим backoff (8 спроб: 1 хв → 5 → 30 → … до доби).
- Rate limit + журнал запитів (логуються й невідомі токени — для виявлення аномалій).
- GeoDB: роздача MaxMind-бази центр → bridge → плагін для гео-резолву на сайтах.

**Безпека.** HMAC на ingest, видачі й пінгах; **fail-closed** — без секрета підпису дані не віддаються (serve → 500, пінг → виключення), а не підписуються порожнім ключем. Один сайт = один read-only токен лише на свої дані. Rate limiting + журнал, аномалії → інцидент.

## 3. WP-плагін DBManager

**Роль.** Read-only клієнт, що ставиться **в кожен сайт WordPress**. Це **не** окремий наш сервер — це плагін на чужих сайтах. Тягне дані з DataBridge по токену, кешує у `wp_options`, рендерить суто PHP (без JS), має mu-plugin фолбек. Редагування значень — лише в Core.

**Де живе.** На кожному сайті WordPress (у проді 50+ реальних сайтів, кожен зі своїм доменом). Для розробки піднято **один** зразковий WordPress як стенд: http://localhost:8080 (вхід WP `admin`/`admin` → меню «DBManager»). Тобто :8080 — «де живе плагін для тесту», а не четвертий продукт.

**Стек.** PHP, WordPress; WP-незалежна логіка покрита PHPUnit з фейками. Запуск стенду — у Docker Compose. Код: DBManager Client.

**Що вже зроблено (план 1.3, з гео на PHP):**
- Доставка: приймач пінга `/wp-json/dbm/v1/ping` (HMAC), fetch свіжих даних, кеш у `wp_options`, добова звірка `If-None-Match`.
- Кеш переживає деактивацію/видалення плагіна; повне очищення — лише окремою кнопкою.
- Вивід: шорткод `[dbm key="phone_ua_1"]` + функція `dbm_get('phone_ua_1')` + `format=tel`; рендер завжди з кешу.
- 3 вкладки wp-admin: «Дані» (read-only), «Вставка» (готові сніпети для WP-менеджера), «Налаштування» (URL bridge, токен, перевірка з'єднання).
- mu-plugin фолбек-рендерер: продовжує рендерити шорткоди/функції з кешу навіть коли основний плагін видалено.
- Гео на PHP: видимість за країною CF-IPCountry → MaxMind → WORLD; вбудований чистий-PHP `maxmind-db/reader`, база роздається з центру через bridge.

**Безпека.** HMAC-перевірка підпису даних від Bridge; робота з кешу при недоступному Bridge (живучість безстроково). **Анти-fingerprint (пріоритет №1)**: у публічному HTML нуль маркерів системи — нейтральні налаштовувані CSS-класи й ім'я шорткода, без data-атрибутів, generator-тегів, унікальних шляхів ассетів і будь-яких зовнішніх ресурсів.

## Чому саме так: ізоляція

Продукт = **Core (CRM)** + **DataBridge (ізоляція)** + **плагін (на сайтах)**. Потік знань односторонній: плагін знає лише URL DataBridge + токен; **Bridge не торкається БД Core**. Тому злам публічного Bridge не розкриває внутрішній центр, а компрометація одного сайта дає доступ лише до його власних даних. Це і є сенс «третього URL». Затверджена архітектура — [[DBManager — Дизайн]] (§3, §9).
