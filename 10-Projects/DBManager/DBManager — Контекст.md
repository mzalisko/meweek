# DBManager — Контекст

**Оновлено:** 2026-06-11
**Стан:** Дизайн затверджено — далі план імплементації ([[DBManager — План]], Етап 1)

**Останні рішення (brainstorm 2026-06-11):**
- Проєкт перейменовано NMDB → **DBManager** (Core = центр, DataBridge = шар ізоляції, DBManager = WP-плагін)
- Повна затверджена спека: [[DBManager — Дизайн]] — джерело правди
- Архітектура A: Core (внутрішня зона) + окремий публічний DataBridge; Laravel + Docker
- Доставка подієва (пінги при змінах/failover, ≤ 1 хв) + добова звірка; без постійного pull
- Слоти номерів з резервами, pin, sticky; вичерпання → типово прибрати вивід
- Гео-мітки (Світ/UA/RU/BY+) — рендер через JS повз page cache
- Плагін: вкладки Дані/Вставка/Налаштування; кеш переживає видалення (mu-plugin фолбек)
- Пріоритети №1: анти-fingerprint (по плагіну не знайти центр) + нуль зовнішніх залежностей
- Без 2FA (рішення користувача); алерти лише в адмінці (дзвіночок до «Опрацьовано»)
- UI: грід-перший, димчасто-синій акцент #54708c, контурні іконки; макети в `.superpowers/brainstorm/`

**Наступний крок:** навичка writing-plans → детальний план Етапу 1 (прототип: Docker, Core-моделі, failover-движок TDD, bridge-контракт, плагін з mu-фолбеком)

**Відкриті питання:** шлях webhook моніторингу (напряму в Core чи приймач на bridge) і VPN vs allowlist — вирішити з DevOps при розгортанні
