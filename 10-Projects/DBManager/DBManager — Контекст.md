# DBManager — Контекст

**Оновлено:** 2026-06-21
**Стан:** Нові типи даних РЕАЛІЗОВАНО: `social` (платформа+нік+url), `address` (структуровано A+), добито `text`. Інтегровано в усі механізми. 271 тест PASS (+12 нових).

## Реалізовано (нові типи)
- **Фундамент:** дозволені типи беруться з `value_types` (`ValueEditor.php` — єдина точка істини замість двох розбіжних `in:`-списків); address/social/text тепер створюються.
- **social:** валідація `network` (enum-select) + handle; content `{value, network, url}`; публікація `socialItem`; мітки/іконки в сітці й аудиті.
- **address (A+):** структуровані поля (`city` обов'язкове) + `value`-дзеркало; `buildAddressContent()`; публікація `addressItem` з allow-list (без витоку службових ключів); bulk заблоковано `set_value/replace_text` (лише `set_geo/set_status`).
- **Підсвічування single-edit:** показ «Було: …» у редакторі; bulk-прев'ю коректне для нових типів завдяки `value`-дзеркалу.
- **Тести:** `tests/Feature/Admin/NewDataTypesTest.php` — усі механізми (create/validate/edit/publish/bulk/audit) + регресія.

## Нові типи даних (соцмережі + адреси) — дослідження
- Робочі типи сьогодні: `phone`, `messenger`, `price`. `address`/`social`/`text` засіяні у `value_types`, але **не підключені** до механізмів (провалюються в generic-текст; create-валідація `ValueEditor.php:146` їх блокує).
- Карта механізмів: [[DATA_TYPES_MAP]] (тип → 8 механізмів, реєстр хардкод-списків, матриця готовності).
- Аудит масових змін: [[BULK_EDIT_AUDIT]] (8 точок плутанини, пріоритети P0–P2, стан підсвічування).
- Рішення «Адреса»: [[ADDRESS_DECISION]] — обрано **A+** (структуровано + `value`-дзеркало); «вільний текст» закриває окремий тип `text`.
- План інтеграції: [[NEW_TYPES_PLAN]] (social ≈ messenger без слотів; address структуровано; контракт тестів на всі механізми).
- Розвідка: фан-аут 8 агентів + критик, перехресно звірено читанням коду. Скрипт: `recon-workflow.js`.

## Наступні кроки (з [[BULK_EDIT_AUDIT]], не зроблено)
- P0 bulk: багате підтвердження (echo операції/кількості/«80 з N»), скоупити `replace_text` з per-field diff.
- P1 bulk: guard проти no-op `set_value`; відокремити apply від publish; повна історія відкату.
- Підсвічування: єдиний diff-компонент для bulk+single; live before/after у single edit.
- Адреса як PII: повноцінний `can_view_address` (окреме, повністю проаудитоване рішення) — поки ungated.
