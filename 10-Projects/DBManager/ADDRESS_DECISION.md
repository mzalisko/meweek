# ADDRESS_DECISION — структурована «Адреса» vs гнучкий текст

> Архітектурне рішення для типу «Адреса». Базується на [[DATA_TYPES_MAP]] (реальна механіка `content` і точок дотику).
> Пов'язано: [[NEW_TYPES_PLAN]], [[DBManager — Контекст]].

## Контекст рішення (факти з коду)

- `content` — нетипізований JSON-blob; жоден шар не валідує його форму.
- Generic-механізми читають **тільки** `content['value']` (`SitePayloadCompiler` default arm `122`, bulk `displayValue:870`, пошук, прев'ю).
- Bulk-операції `set_value`/`replace_text` працюють **лише** по `content['value']` / по всьому JSON — структуровані під-поля вони **не бачать** коректно.
- Тип `text` **вже засіяний** (`ValueTypeSeeder.php:18`) і покриває потребу «просто текст» — його лишилось довести до робочого стану (зняти create-блок, додати label/іконку).

## Варіант A — структурований тип «Адреса»

`content = {country, region, city, street, postcode, value}`, де `value` — **похідний** зібраний рядок («м. Київ, вул. Хрещатик 1, 01001»), що тримається в синхроні.

**Плюси**
- Істина структурована → можлива per-field валідація, майбутній структурований пошук/фільтр, локалізоване форматування на сайті.
- Публікація віддає реальні поля (`buildItem` default уже розгортає всі ключі `content`) → сайт отримує `{country, city, street, …}`, а не сирий рядок.
- Логування/аудит можуть показувати зрозумілий per-field diff (street змінено, місто ні).
- `value`-дзеркало тримає сумісність із generic-машинерією: bulk-прев'ю, пошук, payload показують осмислений рядок, а не `'—'`.

**Мінуси**
- Більше точок дотику: окремий arm у `buildItem`, per-field поля в редакторі, per-field diff в аудиті, гілка в `displayValue`.
- Bulk `set_value` по структурованій адресі небезпечний → треба явно або заблокувати, або зробити типову bulk-операцію.
- Потрібна логіка sync `value`-дзеркала при кожному записі.

## Варіант B — гнучкий текстовий тип (текст ± парсинг у адресу)

`content = {value: "вільний текст"}`, опційно парситься у структуру.

**Плюси**
- Майже нуль нових точок дотику — `content['value']` уже всюди працює (заблоковано лише create-валідацією).
- Простий редактор (одне поле), просте bulk-редагування й пошук по підрядку.

**Мінуси**
- **Немає структури** → немає валідації, структурованого пошуку/фільтра, локалізованого форматування.
- Парсинг «вільний текст → структура» **ненадійний і лоссі**: немає address-парсера, будувати його — поза скоупом; результат недетермінований.
- Сайт отримує сирий рядок — менша цінність для «менеджера БД», чия суть — структуровані, фільтровані дані.
- По суті **дублює** вже наявний тип `text`.

## Порівняння під ключові механізми

| Критерій | A (структура) | B (текст) |
|---|---|---|
| Масові зміни | потребує типової операції, але безпечно/прозоро | «працює», але як до opaque-тексту; `set_value` коректний |
| Логування/diff | per-field, зрозуміло | лише рядок «старе→нове» |
| Пошук/фільтр | структурований (перспектива) | лише підрядок |
| Валідація | per-field (місто/індекс обов'язкові тощо) | лише «не порожньо» |
| Точки дотику | більше (~6) | мінімум (~2) |
| Дублювання з `text` | немає | **повне** |

## РІШЕННЯ (рекомендація)

**Обрати A — структурований тип «Адреса» з похідним `value`-дзеркалом («A+»).**

Обґрунтування:
1. DBManager — менеджер **структурованих, фільтрованих** даних, що публікуються на сайти; вільний текст суперечить суті продукту.
2. `value`-дзеркало знімає головний мінус A: усі generic-механізми (bulk-прев'ю, пошук, payload, аудит-fallback) лишаються осмисленими **без** ризику `'—'`, поки структура дає додаткову цінність.
3. Потреба «просто текст» **уже** закрита окремим первокласним типом `text` (засіяний, лишилось добити) — тож, обираючи A для адреси, ми **нічого не втрачаємо** з гнучкості B; ми просто не вантажимо адресу подвійною роллю.
4. Ненадійний парсинг вільного тексту в адресу виноситься з критичного шляху: введення — структуроване від початку.

**Канонічна форма:**
```
address.content = {
  country:  string|null,
  region:   string|null,
  city:     string,        // обов'язкове
  street:   string|null,
  postcode: string|null,
  value:    string         // похідний зібраний рядок (sync при записі)
}
```

**Bulk-політика для адреси (узгоджено з [[BULK_EDIT_AUDIT]] B1):** на старті дозволити для адреси лише `set_status`/`set_geo` (структурно безпечні), а `set_value`/`replace_text` — приховати/заблокувати, поки не буде типової address-операції. Це уникає тихої псування структурованих полів.

> Якщо знадобиться суто вільний текст — використовувати тип `text`, а не перевизначати `address`.
