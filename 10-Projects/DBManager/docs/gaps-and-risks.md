# Аналіз ризиків, розбіжностей та шляхи оптимізації DBManager

**Дата аудиту:** 2026-06-19
**Модель аудиту:** Gemini (Advanced Agentic Coding)

Цей документ класифікує виявлені архітектурні, технічні, безпекові та користувацькі ризики системи DBManager та пропонує конкретні рішення для їх усунення.

---

## 1. Архітектурні слабкі місця (Architectural Gaps)

1. **Єдина точка відмови (Single Point of Failure)**:
   - **Ризик**: DataBridge є єдиною точкою доступу для всіх сателітів. Якщо сервер DataBridge падає, жоден з 50+ WP-сайтів не зможе синхронізувати оновлені телефони чи ціни.
   - **Пом'якшення**: Сателіти продовжують обслуговувати трафік за допомогою персистентного кешу в `wp_options`. Проте затримка оновлень при вичерпанні лімітів ліній (failover) стає критичною.
2. **Пасивне відстеження працездатності (Health Check)**:
   - **Ризик**: Адмін-панель Core показує час останньої активності сайту (`last_seen_at`) на основі вхідних pull-запитів. Якщо сателіт повністю перестав оновлюватися через помилки cURL або ліміти, Core про це не знає, доки черга `DeliverPingJob` не зафіксує помилку.

---

## 2. Security-ризики (Security Risks)

1. **Атаки повторення (Replay Attacks) на HMAC**:
   - **Ризик**: Будь-який перехоплений запит публікації або пінгу може бути відправлений зловмисником повторно, оскільки підпис (`X-Signature`) не містить фактору часу.
   - **Вплив**: Зловмисник може примусово відкотити версію сайту назад.
2. **Відсутність валідації HTTPS для API-адрес**:
   - **Ризик**: Введення `http://` замість `https://` при налаштуванні `bridge_url` призведе до витоку `X-Site-Token` та незашифрованого трафіку з телефонами.
3. **Відсутність лімітування (Rate Limiting) роуту пінгу на WP-сайті**:
   - **Ризик**: WordPress endpoint `POST /wp-json/dbm/v1/ping` не має обмеження на кількість запитів.
   - **Вплив**: Зловмисник може завалити WP-сайт запитами, що призведе до DoS через постійне ініціювання pull-запитів на DataBridge.

---

## 3. UX-проблеми адмінки (UX Issues)

1. **Візуалізація перекриття (Overrides)**:
   - У гріді значень [ValuesGrid.php](file:///c:/Dev/Meweek/10-Projects/DBManager/code/core/app/Livewire/ValuesGrid.php) відсутній чіткий індикатор того, успадковане значення від групи чи створено індивідуальне перекриття для цього сайту. Це може заплутати контент-менеджерів.
2. **Ручне керування пріоритетами номерів**:
   - Поточний інтерфейс телефонного слота використовує кнопки вгору/вниз (↑/↓) для переміщення номерів у черзі. Це незручно при роботі з великою кількістю номерів (більше 5).
3. **Ручне встановлення MU-Plugin Fallback**:
   - Користувачу доводиться самостійно копіювати файл `dbmanager-fallback.php` та `render-core.php` у папку `mu-plugins`.

---

## 4. Проблеми з даними та синхронізацією (Data & Sync Gaps)

1. **Запізнення видалення токенів (Revocation Gap)**:
   - При видаленні токена або сайту в Core, запис на DataBridge не видаляється моментально, оскільки Core не надсилає DELETE-запит. Токен лишається валідним на зчитування старої версії даних.
2. **Конкурентні оновлення версій (Version Concurrency)**:
   - Якщо два адміністратори одночасно зберігають зміни в Core, може виникнути колізія версій. Хоча DataBridge відхилить застарілу версію (статус 409), Core не має механізму автоматичного авто-інкременту або попередження про конфлікт для другого адміністратора.

---

## 5. Проблеми з тестуванням (Testing Gaps)

1. **Відсутність наскрізних (E2E) інтеграційних тестів**:
   - Наявні тести покривають окремо Core ([PriceSlotTest.php](file:///c:/Dev/Meweek/10-Projects/DBManager/code/core/tests/Feature/Admin/PriceSlotTest.php)), DataBridge та WP-Plugin.
   - **Відсутній** повний тест сценарію: `Зміна в Core` → `Публікація на Bridge` → `Пінг на WP` → `Pull з WP` → `Рендеринг на сторінці WP` в межах одного тестового запуску.
2. **Взаємний вплив тестів через кеш блокувань (EditLock)**:
   - **Ризик**: Механізм спільного редагування (EditLock) використовує Laravel Cache для збереження стану блокувань. PHPUnit не очищує Cache автоматично між окремими тестами в межах одного процесу.
   - **Вплив**: При запуску всього пакету тестів (`php artisan test`) блокування з попередніх тестів перетікають у наступні, викликаючи помилки доступу на Livewire компонентах. Окремо запущені класи при цьому проходять успішно.
   - **Рішення**: Додати `Cache::flush()` у метод `tearDown()` базового класу тестів `Tests\TestCase` або перед тестами, які тестують редагування слотів.

---

## 6. Проблеми масштабування (Scaling Risks)

1. **Зростання бази запитів на DataBridge**:
   - Таблиця `request_logs` (middleware `bridge.log`) збирає записи про кожен запит сателітів. За кілька місяців роботи 50+ сайтів вона виросте до мільйонів рядків, що сповільнить роботу MySQL.
2. **Файлові блокування MaxMind GeoIP**:
   - Метод `put` у [FileGeoDbStore.php](file:///c:/Dev/Meweek/10-Projects/DBManager/code/plugin/dbmanager/src/Geo/FileGeoDbStore.php#L26) перезаписує файл бази `.mmdb` під час синхронізації з використанням блокування `LOCK_EX`. При високому трафіку на сайті, одночасне читання файлу в `MaxMindCountryLookup` може викликати тимчасове блокування потоків веб-сервера.

---

## 7. Рішення для оптимізації (Optimization Solutions)

### 7.1. Кешування інстансу MaxMind Reader
- **Поточний стан**: У [MaxMindCountryLookup.php](file:///c:/Dev/Meweek/10-Projects/DBManager/code/plugin/dbmanager/src/Geo/MaxMindCountryLookup.php#L11) на кожен виклик створюється новий об'єкт `Reader` та закривається:
  ```php
  $reader = new Reader($this->dbPath);
  $record = $reader->get($ip);
  $reader->close();
  ```
- **Оптимізація**: Кешувати відкритий інстанс `Reader` в межах життєвого циклу запиту PHP (наприклад, як статичну або синглтон-властивість класу), щоб уникнути постійного зчитування та парсингу заголовків файлу `.mmdb` з диска при багаторазовому виклику `dbm_get()`.

### 7.2. Ротація логів DataBridge
- Додати заплановану команду `dbm:clean-logs` в Laravel scheduler, яка виконуватиме:
  ```php
  RequestLog::where('created_at', '<', now()->subDays(14))->delete();
  ```
  і запускати її щодня.

### 7.3. Автоматизація встановлення Fallback
- При активації плагіна у [Plugin.php](file:///c:/Dev/Meweek/10-Projects/DBManager/code/plugin/dbmanager/src/Wp/Plugin.php#L45) автоматично перевіряти наявність папки `wp-content/mu-plugins/` та копіювати туди файли `dbmanager-fallback.php` і `render-core.php` з папки плагіна, повідомляючи адміністратора про успішне налаштування аварійного режиму.
