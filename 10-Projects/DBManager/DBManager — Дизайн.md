# DBManager — Дизайн системи

Затверджений дизайн за результатами сесії brainstorming 2026-06-11. Джерело правди для плану імплементації.

Проєкт: [[DBManager — Огляд]] · Вимоги: [[DBManager — Вимоги]] · Безпека: [[DBManager — Безпека]]

## 1. Іменування

| Компонент | Назва | Примітка |
|---|---|---|
| Центральна система | **DBManager Core** | адмінка + БД + failover-движок; внутрішня зона |
| Ізоляційний шар | **DataBridge** | внутрішня назва, назовні не світиться |
| WP-плагін | **DBManager** (код: DBManager Client) | єдине, що бачать на сайтах |

**Наскрізні пріоритети №1:** (а) по плагіну/HTML неможливо відстежити центральну систему; (б) нуль зовнішніх залежностей — без CDN, зовнішніх шрифтів/скриптів/бібліотек у runtime; все self-hosted.

## 2. Масштаб і користувачі

- 50+ WP-сайтів на старті, до ~500 у перспективі. Сервіс одного відділу.
- Ролі: **суперадмін** (все), **менеджери 2–4** (права ріжуться за сайтами/групами та за діями), **в'ювери** (лише перегляд).
- Стек центру: **PHP / Laravel** (спільна мова з WP-екосистемою). Деплой: **Docker Compose** (центр + bridge + MySQL + Redis) — інфраструктурно-агностично, розгортає DevOps-відділ.

## 3. Архітектура (варіант A — затверджено)

```
ВНУТРІШНЯ ЗОНА (VPN/allowlist)          ПУБЛІЧНА ЗОНА
┌─────────────────────────┐   публікує   ┌──────────────────┐  пінг; дані   ┌─────────────┐
│ DBManager Core (Laravel)│ ───────────► │ DataBridge       │ ────────────► │ WP-сайти    │
│ адмінка·БД·failover·    │              │ (мін. Laravel)   │ ◄──────────── │ плагін      │
│ аудит·імпорт·інциденти  │              │ read-only копія  │  звірка 1/добу│ DBManager   │
└───────────▲─────────────┘              │ rate limit·черга │               └─────────────┘
            │ webhook «номер впав»       └──────────────────┘
   Зовнішній моніторинг номерів (окремий секрет)
```

- **Односторонній потік знань**: плагін знає лише URL bridge + токен; bridge не має доступу до БД центру (центр пушить опубліковані дані в bridge); злам нижньої ланки не розкриває верхню.
- Webhook моніторингу йде в центр (або через приймач на bridge, якщо моніторинг не дістає внутрішньої зони — вирішується з DevOps при розгортанні).
- Bridge автономний при падінні центру; сайти автономні при падінні bridge (кеш).
- З ростом bridge виноситься на окремий хост без зміни коду.

## 4. Модель даних

**Сутності:** Site (сайт, токен, група, країна-орієнтація) · SiteGroup (бренд/сімейство доменів: domen.org+domen.ua) · Value (тип, ключ, вміст, гео-мітки, область: група або сайт-override) · ValueType (phone/messenger/price/address/social/text — розширюваний) · GeoTag (Світ/Україна/Росія/Білорусь — розширюваний) · Token · User+Role · AuditLog · Incident.

**Ключові рішення:**
1. **Спільні значення без дублів**: однаковий номер зберігається один раз, сайти/групи посилаються на нього → bulk-зміна оновлює всіх, хто посилається, одним рухом.
2. **Групи + перекриття**: значення задаються на групі; сайт може перекрити окреме значення власною версією (бейдж «✎ цього сайта»).
3. **Гео-видимість**: кожне значення несе мітки країн «де показувати»; «Світ» — фолбек. Контроль «кому не показувати» — відсутністю мітки.

**Телефонні слоти і failover (серце системи):**
- **PhoneSlot** — логічна позиція (`phone_ua_1`); сайт бачить ключ слота. На сайт може бути кілька слотів на країну (2 UA, 3 RO…).
- **NumberEntry** — фізичний номер у слоті з пріоритетом (#0 основний, #1..#n резерви), статус active/down.
- Слот показує перший за пріоритетом активний номер. Номер ожив → авто-повернення (опційно **sticky** — лишатись на резерві до ручного повернення).
- **Ручне закріплення (pin)**: менеджер може примусово показувати конкретний номер; авто-перемикання вимкнене до зняття pin.
- **Вичерпання** (всі down): політика на рівні слота — типово **повністю прибрати вивід**; варіанти: показувати останній / аварійний номер. Завжди — інцидент.
- **Месенджери двох видів**: прив'язані до слота (Viber/WhatsApp — слідують за активним номером при failover) та незалежні (Telegram-канал). У гріді — окремими рядками.

## 5. Доставка даних (подієва, затверджено)

1. **Тригер**: збереження зміни / bulk / failover → центр компілює готові дані уражених сайтів → пушить у bridge → bridge шле push-пінги лише ураженим сайтам → плагін одразу тягне свіже. На сайтах — за секунди (вимога failover: **≤ 1 хв**).
2. **Ретраї**: недоставлений пінг повторюється з наростаючим інтервалом (1 хв → 5 → 30 → … до доби).
3. **Страхувальна звірка 1/добу**: плагін питає «моя версія N актуальна?» (~200 байт). Ловить загублені пінги + працює пульсом парку: сайт мовчить кілька днів → інцидент.
4. Постійного опитування (poll кожні 15 хв) **немає** — за рішенням користувача.

## 6. Webhook моніторингу

- Джерела сигналу «номер неактивний»: зовнішня система моніторингу (webhook з окремим секретом) + ручне перемикання в адмінці. Вбудованого моніторингу номерів немає.
- Захист від failover-шторму: rate limit на webhook + зведений інцидент «масовий failover» замість сотень окремих.

## 7. Адмінка DBManager Core (UI затверджено візуально)

- **Каркас**: дата-центричний грід (варіант C) — головний екран = потужна таблиця значень.
- **Контекст**: хлібні крихти `Усі сайти › Група › Сайт` + смужка контексту («Ви працюєте з сайтом domen.ua…», лінк повернення). Область дії в кожному рядку (Група / ✎ цього сайта).
- **Структура**: бічна навігація (Дашборд / Значення / Сайти і групи / Інциденти / Аудит / Доступи й токени); грід згруповано за типами з лічильниками; фільтри (тип/гео/статус/пошук); bulk-дії над виділеним з прев'ю «торкнеться N сайтів» і відкатом пакета.
- **Панель слота** (праворуч, без виходу зі списку): гео-мітки, область дії, ланцюг номерів (drag-пріоритети, статуси, «Показувати цей» = pin), прив'язані месенджери, режим повернення (Авто/Sticky), поведінка при вичерпанні, Зберегти → аудит + push.
- **Дашборд**: компактна смужка стану (варіант A: онлайн X/Y · на резерві · вичерпано · змін сьогодні · остання синхронізація) → список «Потребує уваги» → стрічка аудиту з кнопкою «відкотити».
- **Дзвіночок інцидентів**: інцидент «новий», доки не натиснуто «Опрацьовано ✓» — переживає закриту адмінку; опрацьовані сіріють в історії. Алерти — лише в адмінці (без Telegram/email), за рішенням користувача.
- **Стиль**: світлий нейтральний фон (#f4f5f3), білі картки, **димчасто-синій акцент #54708c**, контурні іконки (Lucide-стиль), статуси на пастельних плашках (шавлія/бурштин/цегла). Макети: `.superpowers/brainstorm/*/content/grid-failover-v5.html`, `dashboard.html`.
- Адмінка українською. Імпорт стартових даних: CSV/XLSX + ручне добирання (дані зараз частково в таблицях, частково на сайтах).

## 8. WP-плагін «DBManager»

**Сторінки wp-admin:**
1. **«Дані»** (read-only): все отримане — слоти з повним ланцюгом резервів і статусами, гео-варіанти, мітки; стан зв'язку (остання синхронізація, версія, «зв'язок втрачено з… — показуються збережені дані»). Редагування значень — лише в центрі.
2. **«Вставка»** (панель для WP-менеджера): список ключів з поточними значеннями, готові сніпети з копіюванням — шорткод `[dbm key="phone_ua_1"]`, `dbm_get('phone_ua_1')`, варіант з `tel:`; пошук/фільтр.
3. **«Налаштування»** (лише адмін): URL bridge, токен, перевірка з'єднання, CSS-префікс виводу та ім'я шорткода (нейтральні, налаштовувані).

**Живучість даних (вимога користувача):**
- Кеш у `wp_options`; **не видаляється** при деактивації/видаленні плагіна (повне очищення — лише окремою кнопкою).
- **mu-plugin фолбек-рендерер**: ставиться разом з основним плагіном у `mu-plugins`, переживає видалення звичайних плагінів, продовжує рендерити шорткоди/функції з кешу до відновлення плагіна.
- Втрата зв'язку → сайт показує останні дані безстроково; центр бачить пропущені звірки → інцидент.

**Вивід і гео:**
- Рендер завжди з кешу (нуль звернень до bridge при показі сторінок).
- Гео-залежні значення — через JS повз кеш сторінок (на більшості сайтів є page cache): нейтральний span-плейсхолдер з дефолтом «Світ» (анти-мерехтіння) → один AJAX-запит до свого ж сайту → сайт визначає країну (заголовок Cloudflare → локальна MaxMind-база, яку роздає центр через bridge) → підставляються правильні варіанти. Значення без гео-варіантів рендеряться одразу в PHP.
- **Анти-fingerprint (пріоритет №1)**: у публічному HTML нуль маркерів системи — нейтральні налаштовувані CSS-класи/шорткод, без data-атрибутів, generator-тегів, коментарів, унікальних шляхів ассетів. Жодних зовнішніх ресурсів.

**Розгортання**: вручну на старті; власний сервер оновлень з центру — окремий етап. Підключення сайту: створити сайт в адмінці → згенерувати токен → вставити в налаштування плагіна.

## 9. Безпека (затверджено; деталі — [[DBManager — Безпека]])

- Три зони довіри: центр невидимий з інтернету (VPN/allowlist) → bridge публічний але «порожній» (лише опубліковані дані) → сайти знають мінімум.
- Токени: один сайт = один read-only токен лише на свої дані; відкликання за секунду; ротація.
- HMAC-підписи на пінгах і відповідях з даними; увесь трафік HTTPS.
- Вхід у центр: пароль **без 2FA** (рішення користувача); компенсатори — мережева ізоляція адмінки, ліміт спроб входу, аудит входів.
- Webhook моніторингу — окремий секрет. Rate limiting + журнал запитів на bridge, аномалії → інцидент.
- Секрети в `.env` поза git; екранування виводу (XSS); бекапи БД центру щодня + перевірений сценарій відновлення; bridge не бекапиться (перепублікація з центру).

## 10. Відмови (визначена поведінка)

| Сценарій | Реакція |
|---|---|
| Bridge недоступний сайту | кеш безстроково; ретраї звірки; інцидент у центрі |
| Центр упав | bridge віддає дані автономно; черга надолужує після підняття |
| Пінг не доставлено | ретраї до доби → добова звірка → інцидент якщо сайт мовчить днями |
| Битий webhook/підпис | відхилення + журнал; повторювані фейли → інцидент |
| Failover-шторм | rate limit + один зведений інцидент |
| Битий CSV-імпорт | все або нічого + звіт помилок по рядках |
| Конкурентне редагування | оптимістичне блокування («значення щойно змінили») |
| Помилковий bulk | відкат пакета однією дією з аудиту |

## 11. Аудит

Повний журнал: хто/коли/що, старе → нове, джерело (людина / webhook / failover-движок), bulk — одним пакетом. Відкат одиничної зміни і bulk-пакета. Стрічка на дашборді.

## 12. Тестування

Unit (failover-движок, гео-резолвер, RBAC) · контрактні тести API DataBridge (плагін ↔ bridge не розходяться; контракт = можливість переносити центр без оновлення плагінів) · тести плагіна (рендер, кеш переживає видалення, mu-plugin без основного плагіна, «bridge мертвий → сайт працює») · E2E failover із заміром ≤ 1 хв · навантаження bridge (500 сайтів) · безпековий чекліст (чужий/відкликаний токен, битий HMAC, rate limit).

## 13. Поза скоупом MVP (пізніші етапи)

Блоки Gutenberg · власний сервер оновлень плагіна · зовнішні канали алертів (Telegram/email) · 2FA (опційно) · фізичне рознесення центру і bridge по різних хостах.
