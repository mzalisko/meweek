# DBManager — Вимоги

Усі питання етапу уточнення закрито 2026-06-11 (сесія brainstorm). Деталі рішень — [[DBManager — Дизайн]].

Проєкт: [[DBManager — Огляд]]

## Функціональні вимоги

1. **Центральне сховище даних**: телефони (слоти з резервами), месенджери (прив'язані до номерів та незалежні), ціни, адреси, посилання на соцмережі, довільні значення.
2. **Гео-таргетинг**: кожне значення має мітки країн «де показувати» (база: Світ/Україна/Росія/Білорусь, розширювана); плагін показує варіант за країною відвідувача.
3. **Failover номерів**: слот має ланцюг резервів; сигнал «номер впав» (webhook зовнішнього моніторингу або вручну) → перемикання на наступний живий резерв → на сайтах ≤ 1 хв. Ручне закріплення (pin), sticky-режим, налаштовувана поведінка при вичерпанні (типово — прибрати вивід).
4. **Адмінка** з дата-центричним грідом, bulk-діями (з прев'ю та відкатом), імпортом CSV/XLSX, дашбордом і дзвіночком інцидентів (висять до «Опрацьовано»).
5. **Доставка подієва**: тригер при збереженні/bulk/failover (push-пінг ураженим сайтам) + добова страхувальна звірка. Без постійного опитування.
6. **Підтримка 50–500 сайтів**: групи (сімейства доменів: domen.org+domen.ua) зі спільними значеннями + перекриття на рівні сайта; спільні значення без дублів (посилання) для bulk-дій.
7. **Ролі**: суперадмін, 2–4 менеджери (права за сайтами/групами + за діями), в'ювери.
8. **Повний аудит з відкатом** (включно з bulk-пакетами і авто-failover).
9. **WP-плагін**: вкладки «Дані» (read-only перегляд отриманого з резервами/гео/статусами), «Вставка» (сніпети для менеджера), «Налаштування»; шорткоди + PHP-функції.

## Архітектурні вимоги

1. **DataBridge — ізоляційний шар**: плагін знає лише адресу/контракт DataBridge; центр для плагіна не існує. Контракт API дозволяє переносити/міняти центр без оновлення плагіна.
2. **Анти-fingerprint (пріоритет №1)**: по плагіну і його HTML неможливо відстежити центральну систему — нейтральні налаштовувані CSS-класи/шорткод, нуль маркерів у публічному виводі.
3. **Без зовнішніх залежностей (пріоритет №1)**: жодних CDN, зовнішніх бібліотек/шрифтів/скриптів/API у runtime; все self-hosted.
4. **Живучість**: сайт рендерить з кешу завжди; дані переживають втрату зв'язку, деактивацію і видалення плагіна (mu-plugin фолбек) — до відновлення.
5. **Інфраструктурна агностичність**: Docker Compose, розгортає DevOps будь-де; центр у внутрішній зоні (VPN/allowlist), bridge публічний.

## Нефункціональні вимоги

1. Безпека — [[DBManager — Безпека]] (без 2FA — рішення користувача).
2. Продуктивність: відповідь bridge не впливає на швидкість сайтів (рендер з кешу; гео — один AJAX-запит до свого сайту повз кеш сторінок).
3. Стек: PHP/Laravel (центр і bridge), MySQL, Redis. Адмінка українською.

## ✅ Відповіді на питання уточнення (2026-06-11)

- Сайтів: 50+ на старті, до ~500. Один відділ; суперадмін + 2–4 менеджери (різні права) + в'ювери.
- Хостинг: є DevOps-відділ, деталі невідомі → інфраструктурно-агностичний Docker.
- Доставка: подієва (тригери при змінах/failover), без 15-хв pull; добова звірка як страхувальна сітка.
- Failover до сайтів: ≤ 1 хв. Джерела сигналу: зовнішній моніторинг (webhook) + вручну.
- Стек: PHP/Laravel. Аудит: повний + відкат.
- Вивід: шорткоди + PHP-функції тем (блоки Gutenberg — пізніше). Кастомна адмінка, спроєктована разом (грід-перший, димчасто-синій акцент).
- Гео: сайти за Cloudflare (заголовок) + MaxMind-фолбек; на більшості сайтів page cache → гео-значення через JS.
- Месенджери: і прив'язані до номерів (failover разом), і незалежні.
- Вичерпання резервів: налаштовується на слоті, типово — повністю прибрати вивід; алерт завжди.
- Алерти: лише в адмінці (дзвіночок з постійним журналом до «Опрацьовано»).
- Імпорт: дані частково в таблицях, частково на сайтах → CSV/XLSX імпорт + ручне добирання.
- Плагін: встановлення вручну на старті; авто-оновлення — окремий етап.
