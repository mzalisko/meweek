# skill-forge — кузня навичок

Двигун, що веде навичку через життєвий цикл: **ідея → готовий скіл → запис у людському гайді → (далі) самопокращення**. Це єдине джерело правил «як ми в Meweek робимо й документуємо скіли».

## Режим A. Ідея → скіл  («впровадь <назва ідеї>»)

1. Прочитай файл ідеї в `30-Agents/Skill Ideas/`.
2. Оціни чесно: цінність, складність, безпека; чи не дублює наявний скіл, superpowers або `claude-api`.
3. Якщо **не варте уваги або ризиковане** — не будуй. Напиши чому і що робити натомість (лишити ідеєю, спростити, винести в рутину).
4. Якщо варте:
   - Створи `.claude/skills/<name>/SKILL.md` — frontmatter (`name` англійською, `description` українською), тіло — короткі кроки.
   - Створи тонку команду `.claude/commands/<name>.md` — вказівник на SKILL.md без дубльованої логіки.
   - Додай запис у `50-Guides/Гайд — Навички Meweek простими словами.md` — людською мовою: що це, коли кликати, як покликати, приклад.
   - Якщо створена команда, онови `50-Guides/Гайд — Команди Meweek простими словами.md`.
   - Онови `30-Agents/Конектори і навички.md`, якщо змінився перелік власних навичок або команд.
   - Познач ідею як реалізовану: банер `> ✅ Реалізовано <дата> → <скіл>` угорі файла ідеї (не видаляй — це провенанс).
   - Пройди `30-Agents/Перевірка документації.md`: перевір людський гайд, агентний контракт, індекси й пошук по назві.
   - Коміт українською: `навички: впроваджено <name>`.

## Режим B. Самопокращення  («покращ <name>», «розвинь скіли»)

1. Візьми наявний `SKILL.md`.
2. Звір із практикою: що збоїло в реальному використанні, що додав користувач у фідбеку, чи актуальні факти (моделі, шляхи, прапорці — **перевір, що вони ще існують**, перш ніж радити).
3. Внеси точкове покращення. Не переписуй заради переписування.
4. Запиши зміну в розділ `## Журнал змін` у кінці SKILL.md (дата + одне речення — що і чому).
5. Онови запис у людському гайді, якщо змінилась поведінка.
6. Пройди `30-Agents/Перевірка документації.md`.
7. Коміт українською: `навички: покращено <name>`.

## Правила

- Скіли тонкі; контракти проєктних команд живуть у файлах проєкту (AGENTS.md §7), не в скілі.
- **Запобіжник:** критичний, незворотний або мережевий скіл (push, видалення, секрети, все навколо DBManager) — спершу пропозиція і підтвердження користувача, не авто-білд.
- Кожен скіл мусить мати запис у людському гайді. Без нього скіл «невидимий» для людини.
- `50-Guides/` пишеться для користувача, не для агента: коротко, без внутрішніх промптів і довгих чеклістів.
- Не чіпай конфіги інших агентів (`.codex/`, `.gemini/`) без прохання.
- Для самого ремесла створення та евалу скілів є Anthropic-навичка `skill-creator` — використовуй її як інструмент, не дублюй.
- **Крос-агентність:** інші агенти (Codex, Gemini) реалізують можливості своїм механізмом, але за тими самими правилами — службова логіка в `30-Agents`, тонка обгортка на агента, обов'язковий запис у людському гайді. Службовий реєстр для агентів — `30-Agents/Конектори і навички.md`; людська шпаргалка — `50-Guides/Гайд — Навички Meweek простими словами.md`.
