# Docker під проєкт

Мета: швидкі, стабільні контейнери під кожен проєкт; конфігурація — у папці проєкту; секрети — поза git.

## Структура (на проєкт)

```
10-Projects/<Назва>/code/
├── docker-compose.yml      ← сервіси проєкту (в git)
├── Dockerfile              ← якщо є власний образ (в git)
├── .dockerignore           ← (в git)
├── .env.example            ← шаблон змінних БЕЗ значень (в git)
└── .env                    ← реальні доступи (НЕ в git, у .gitignore)
```

## Правила швидкості (чому так)

1. **Slim/alpine базові образи** (`php:8.3-fpm-alpine`, `node:22-alpine`) — менше тягнути й сканувати.
2. **Multi-stage build** — залежності збираються в окремому stage, фінальний образ малий.
3. **.dockerignore обов'язково** (node_modules, .git, vendor) — інакше контекст збірки роздувається і збірка повільна.
4. **Порядок шарів**: спочатку COPY файлів залежностей (package.json/composer.json) + install, потім COPY коду — кеш шарів переживає зміни коду.
5. **Named volumes для БД** — дані переживають перестворення контейнера.

## Правила стабільності (без падінь)

1. `restart: unless-stopped` для всіх сервісів.
2. **healthcheck** кожному сервісу + `depends_on: condition: service_healthy` — застосунок не стартує раніше за готову БД.
3. **Ліміти ресурсів** (mem_limit) — один контейнер не покладе Docker Desktop.
4. Версії образів пінити (`postgres:16.3`, не `latest`) — оновлення тільки свідомі.

## Доступи (credentials)

1. Всі секрети проєкту — лише в `.env` поруч із compose (він у .gitignore глобально).
2. У git — `.env.example` з переліком змінних і коментарями, без значень.
3. compose читає `.env` автоматично (`environment: - DB_PASSWORD=${DB_PASSWORD}`).
4. Жодних секретів у Dockerfile, compose, нотатках чи комітах.

## Кроки створення

1. Спитати: які сервіси (БД? веб? кеш?), які порти зайняті в користувача.
2. Створити файли за структурою вище; для WP-проєктів (NMDB) — компонувати wordpress + mariadb + adminer.
3. Записати в `<Назва> — Огляд.md` команди запуску: `docker compose up -d`, перевірка `docker compose ps`.
4. Перевірити: `docker compose config` (валідність) перед запуском.
