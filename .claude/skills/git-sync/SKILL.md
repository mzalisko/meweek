# Синхронізація git

Мета: стан `C:\Dev\Meweek` завжди актуальний — локально і з будь-якого місця (інша машина, claude.ai через GitHub-конектор).

## На початку сесії (pull)

1. `git status --short` — якщо є незакомічені зміни, спершу спитати користувача (це може бути чужа незавершена робота).
2. Перевірити результати хмарних рутин: `git ls-remote origin "refs/heads/claude/*"` — якщо є гілки, спершу влити їх у main (див. розділ нижче), бо `pull` сам по собі їх не підтягне.
3. `git pull --rebase origin main` — підтягнути віддалені зміни.
4. Якщо конфлікт: НЕ вирішувати мовчки. Показати конфліктні файли користувачу; для .md зазвичай правильно об'єднати обидві версії вручну.

## Результати хмарних рутин (гілки claude/*)

Хмарні рутини claude.ai (наприклад, daily-ai-digest) НЕ комітять у main: кожен запуск пушить окрему гілку `claude/*` і відкриває PR. Поки PR не змерджено, проєкт не оновлено результатами рутини.

1. Знайти гілки: `git ls-remote origin "refs/heads/claude/*"` (PR видно на GitHub).
2. Якщо кілька гілок міняють той самий файл (повторний запуск рутини за той самий день) — це дублі: мерджити найновішу, унікальний корисний вміст зі старішої перенести вручну, її PR закрити без мерджа.
3. Влити в main: змерджити PR на GitHub, або локально `git fetch origin && git merge origin/claude/<гілка>` і запушити.
4. Після мерджа видалити гілку `claude/*` на remote, щоб не накопичувались.
5. Далі — звичайний `git pull --rebase origin main`.

## Наприкінці сесії (commit + push)

1. Оновити контексти проєктів (навичка save-context).
2. `git add -A && git commit -m "розділ: що зроблено"` — коміт українською.
3. `git push origin main` — якщо є credentials. Якщо push недоступний (немає токена в цьому середовищі) — сказати користувачу: «закомічено локально, запуш зі свого термінала: git push».

## Чому rebase, а не merge

Історія середовища лінійна (один користувач, агенти по черзі) — rebase тримає її чистою без merge-комітів.

## Правила

- Ніколи не робити `push --force` на main.
- Ніколи не комітити секрети (.env у .gitignore — перевіряти `git status` перед add).
- Якщо `git status` показує дивні масові зміни (обрізані файли, бита кодировка) — зупинитися і перевірити цілісність (iconv -f UTF-8), перш ніж комітити.
