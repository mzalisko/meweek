# MCP — перехід на stateless протокол

## Що це

Release Candidate специфікації Model Context Protocol (публікація кінець червня 2026, фінал — 28 липня 2026) прибирає `initialize`/`initialized` handshake і заголовок `Mcp-Session-Id` — MCP-сервер більше не тримає stateful-сесію на з'єднання. Додано:

- **Extensions framework** — офіційний механізм розширення протоколу.
- **Tasks** (`tasks/get`, `tasks/update`, `tasks/cancel`) — офіційний статус довготривалих операцій.
- Нові заголовки `Mcp-Method`/`Mcp-Name` для роутингу запитів.
- **Roots, Sampling, Logging — deprecated** (SEP-2577), формальна політика депрекації (SEP-2596, мінімум 12 місяців перед видаленням).

## Чому це важливо

Поточні MCP-сервери й клієнти покладаються на sticky-сесію (один сервер тримає стан з'єднання). Stateless-протокол дозволяє звичайний round-robin load balancer без sticky sessions — простіше масштабувати, але **ламає сумісність** для серверів, написаних під stateful-модель.

Для Meweek це напряму стосується вибору MCP-плагіна для Obsidian: якщо підключити сервер зараз під стару модель, після 28.07.2026 доведеться мігрувати.

## Приклад

До RC: клієнт викликає `initialize`, отримує `Mcp-Session-Id`, усі наступні запити прив'язані до цієї сесії на конкретному інстансі сервера.

Після RC: кожен запит самодостатній (stateless), сервер не мусить пам'ятати попередній контекст з'єднання — балансувальник може направити наступний запит на будь-який інстанс.

## Як використати в Meweek

1. Не обирати остаточний MCP-плагін для Obsidian, поки не підтверджена його сумісність з RC/фінальною специфікацією.
2. Якщо пишемо власний MCP-сервер (наприклад, для проєктних даних) — одразу проєктувати stateless, не покладатись на Roots/Sampling/Logging (deprecated).
3. Слідкувати за фіналізацією 28.07.2026 — на цю дату закладено як контрольну точку в [[AI Дайджест 2026-07-01]].
