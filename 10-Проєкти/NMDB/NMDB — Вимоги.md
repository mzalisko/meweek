# NMDB — Вимоги

Початкова фіксація вимог. Уточнюються в процесі планування — питання для уточнення внизу.

Проєкт: [[NMDB — Огляд]]

## Функціональні вимоги

1. **Центральне сховище даних**: телефони, месенджери (Telegram, Viber, WhatsApp…), ціни, інші довідкові значення.
2. **Адмін-інтерфейс** для швидкої зміни значень з центральної системи.
3. **Доставка даних у WP-плагін** через шар DataBridge: плагін отримує дані й відображає їх на сайті (шорткоди/блоки/віджети).
4. **Швидке поширення змін**: зміна в центрі має з'явитися на сайті без ручних дій (push або короткий кеш).
5. **Підтримка кількох сайтів** з одного центру (різні набори значень для різних сайтів).

## Архітектурні вимоги

1. **DataBridge — ізоляційний шар**: плагін знає лише адресу/контракт DataBridge; центральна система для плагіна не існує. Це дозволяє змінювати/переносити центр без оновлення плагіна.
2. **Без зовнішніх залежностей**: жодних CDN, зовнішніх шрифтів/скриптів, сторонніх API у runtime. Усі ресурси — локальні (self-hosted).
3. **Автономність плагіна**: якщо DataBridge недоступний — сайт показує останні закешовані значення, нічого не ламається.

## Нефункціональні вимоги

1. Безпека — окремий документ: [[NMDB — Безпека]].
2. Продуктивність: відповідь DataBridge не повинна сповільнювати сайт (кешування на стороні плагіна обов'язкове).
3. Простота розгортання: мінімум інфраструктури.

## ❓ Питання для уточнення (до етапу архітектури)

- [ ] Скільки сайтів на старті та в перспективі?
- [ ] Де хоститиметься центральна система (VPS, shared, локально)?
- [ ] Push (webhook у плагін) чи pull (плагін періодично питає DataBridge) — чи комбінація?
- [ ] Хто користувачі центральної системи (тільки ти / кілька осіб з ролями)?
- [ ] Технологічний стек центру (PHP/Laravel? Python? Node?) — чи є переваги?
- [ ] Чи потрібна історія змін значень (аудит)?
