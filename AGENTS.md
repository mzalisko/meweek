# Інструкції для AI-агентів (Meweek)

Цей файл — спільні правила для всіх агентів (Codex, Claude, Gemini та інших), що працюють у цій папці.

## Що це за середовище

`C:\Dev\Meweek` — це одночасно:

1. **Сховище Obsidian** — нотатки, документація, плани (українською).
2. **Робоче середовище проєктів** — код проєктів живе в `10-Projects/<Назва>/code/` або в окремих репозиторіях, на які посилаються нотатки.
3. **Git-репозиторій** — https://github.com/mzalisko/meweek

## Обов'язкові правила

1. **Мова**: вміст усіх `.md` файлів, коміти, коментарі в документації — українською. Назви папок, код і змінні — англійською. Назви `.md` файлів — українською.
2. **Структура**: нові нотатки створювати у правильному розділі (`00-Inbox` для чернеток, `10-Projects/<Назва>/` для проєктних, `20-Knowledge` для постійних знань, `30-Agents` для знань і правил ШІ-агентів, `50-Guides` для людських гайдів користувача). Не створювати файли в корені сховища без потреби.
3. **Посилання**: пов'язувати нотатки через `[[wiki-посилання]]`. Кожна проєктна нотатка має посилатися на огляд свого проєкту.
4. **Шаблони**: для нових проєктів, ADR, зустрічей використовувати шаблони з `_Templates/`. Створення проєкту — за навичкою `.claude/skills/new-project/SKILL.md` (вона ж описує обов'язковий набір файлів проєкту).
5. **Obsidian-граф**: граф будується автоматично з `[[wiki-посилань]]`; покращувати його через змістовні зв'язки між нотатками, а не ручною косметикою. Деталі: `30-Agents/Гайди для агентів/Гайд — Obsidian для Meweek.md` і `30-Agents/Ведення Obsidian.md`.
6. **Не чіпати**: `.obsidian/` (конфіг Obsidian) — редагувати лише на явне прохання.
7. **Git**: осмислені коміти українською у форматі `розділ: що зроблено` (напр. `dbmanager: додано архітектурну схему`). Не пушити без прохання користувача.
8. **30-Agents — службова зона для ШІ**: усе, що агент має знати, пам'ятати, перевіряти або виконувати як внутрішнє правило, класти в `30-Agents/`, `AGENTS.md` або контракт конкретного проєкту. Не ховати агентні правила в `50-Guides/`.
9. **50-Guides — для людини**: у `50-Guides/` тримати короткі користувацькі гайди: що це, коли використовувати, як запустити, як перевірити результат. Не класти туди внутрішні промпти, правила агентів, довгі чеклісти й технічні контракти.
10. **Людський гайд обов'язковий**: кожна нова навичка, команда, важлива фіча або пояснення, яке користувач просить описати, має отримати короткий запис у відповідній папці `50-Guides/` або проєктному `— Як працювати.md`. Якщо користувач каже “запиши в рецепти/AI/код”, дописувати саме в цю людську папку.

## Контекст і токеноміка

1. Чат — одноразовий, файли — постійні: всі рішення одразу фіксувати у файлах проєкту.
2. На старті сесії читати лише `<Проєкт> — Контекст.md` потрібного проєкту, а не все сховище.
3. Перед завершенням сесії / на команду «збережи контекст» — оновити `<Проєкт> — Контекст.md` (≤ 30 рядків).
4. Деталі: `30-Agents/Робота з контекстом.md`.

## Службові директорії

Агенти не читають, не індексують і не додають у контекст службові та згенеровані директорії без прямої потреби задачі. Заборонені за замовчуванням шляхи:

- `**/.git/**`
- `**/vendor/**`
- `**/node_modules/**`
- `**/.venv/**`
- `**/venv/**`
- `**/dist/**`
- `**/build/**`
- `**/.next/**`
- `**/coverage/**`
- `**/storage/**`
- `**/bootstrap/cache/**`

Виняток: якщо проблема прямо стосується залежностей, збірки, кешу, Git-історії або файлу з такої директорії, агент читає лише мінімально потрібні файли й пояснює це в повідомленні користувачу.

Для Obsidian дерево `10-Projects/<Назва>/code/` будь-якого проєкту вважається робочим кодом, а не нотатками. Воно має бути виключене з індексу й графа Obsidian; важливі рішення, огляди й документація дублюються у проєктних `.md`-нотатках поруч із кодом, а не всередині `code/`.

Якщо в будь-якому проєкті в Obsidian потрапляють залежності, кеші, збірки, службові папки або інші згенеровані файли, агент має оновити спільні правила виключення (`.obsidian/app.json`, `.obsidianignore`, `.obsidian/snippets/meweek-hide-service-folders.css`, за потреби цей файл), а не додавати одноразовий фільтр тільки для поточного випадку. Ignore-фільтри чистять індекс/граф, CSS-snippet чистить дерево файлів Obsidian.

## Співіснування агентів (Claude, Codex, Gemini)

1. **Спільна пам'ять = файли + git.** Кожен агент на старті читає цей файл і контекст проєкту — цього достатньо, щоб бути в курсі. Усе зроблене фіксується комітом, тому наступний агент бачить стан через `git log` і файли контексту.
2. **Перед роботою**: `git status` + `git log --oneline -5` — переконатися, що немає чужих незакомічених змін. Якщо є — спитати користувача, не перезаписувати.
3. **Після роботи**: оновити контекст проєкту і закомітити. Незакомічена робота для інших агентів невидима.
4. **Не редагувати конфіги інших агентів** (`.claude/`, `.codex/`, `.gemini/`) без прохання.
5. **Один агент — одна задача за раз** у межах папки; паралельну роботу над одним проєктом не вести.
6. **Рев'ю за ставками**: звичайні зміни — самоперевірка (verification-before-completion); критичний код (DBManager, все, що стосується безпеки чи продакшну) — перед комітом у main обов'язкове рев'ю іншим агентом (крос-модельне: Claude ↔ Codex) + security-review. Команда: `/review`.
7. **Контракти команд проєктів — у файлах проєкту.** Логіка проєктних команд (наприклад, `10-Projects/Fitness/Fitness — Команди.md`) живе у сховищі й спільна для всіх агентів. Обгортки в `.claude/commands/`, `.codex/`, `.gemini/` — тонкі вказівники на контракт без власної логіки; зміни режимів вносити тільки у файл контракту, щоб перемикання між агентами не змінювало поведінку.
8. **Хмарні рутини — теж агенти.** Рутини claude.ai (наприклад, daily-ai-digest) не комітять у main: результати приходять як гілки `claude/*` + PR на GitHub. Під час синхронізації перевіряти ці гілки і вливати їх у main (дублі одного дня — мерджити найновішу, решту закривати), інакше проєкт не отримає результати рутин. Деталі — навичка `.claude/skills/git-sync/SKILL.md`.
9. **Спільні можливості — у службовому реєстрі.** Канонічний перелік навичок/команд для агентів — `30-Agents/Конектори і навички.md`; людська шпаргалка — `50-Guides/AI/Гайд — Навички Meweek.md`. Кожен агент звіряється зі службовим реєстром перед роботою і дописує туди кожну нову можливість. Нова можливість дозріває з ідеї у `30-Agents/Skill Ideas/` за процесом skill-forge: логіка живе у спільному файлі сховища, обгортка на агента тонка, людський гайд обов'язковий, а для критичного/незворотного — спершу підтвердження, не авто-білд.

## Безпека

- Жодних секретів (паролі, API-ключі, токени) у нотатках чи комітах. Доступи кожного проєкту — лише у його `10-Projects/<Назва>/code/.env` (у `.gitignore`); у git — `.env.example` без значень.
- Проєкт DBManager має підвищені вимоги до безпеки — див. `10-Projects/DBManager/DBManager — Безпека.md`.

## Проєкти

Актуальний список — `10-Projects/Проєкти.md`. Перед роботою над проєктом прочитати його `— Контекст.md`, потім `— Огляд.md`.

## Команди середовища

- Fitness у Codex/Gemini: `fitness Микола`, `fitness Марта`, `fitness Микола smoke`, `fitness <ім'я> підбий тиждень`, `fitness <ім'я> імпортуй резервні звіти` — працювати за `10-Projects/Fitness/Fitness — Команди.md`.
- Fitness у Claude: ті самі команди через `/fitness ...` (`.claude/commands/fitness.md` — тонка обгортка, логіка в `Fitness — Команди.md`).
- Вибір моделі під задачу: Claude — `/model-budget <задача>`; службові правила — у `.claude/skills/model-budget/SKILL.md`, людський опис — у `50-Guides/AI/Гайд — Навички Meweek.md`.
- Розвиток навичок (ідея→скіл, покращення): Claude — `/skill-forge впровадь <ідея>` / `покращ <скіл>`; контракт життєвого циклу — `.claude/skills/skill-forge/SKILL.md`, з обов'язковим записом у гайді навичок.
