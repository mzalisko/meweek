# Робота з контекстом і токеноміка

Середовище побудоване так, щоб `/clear` (скидання контексту агента) нічого не втрачав: **вся пам'ять живе у файлах, а не в чаті**.

## Принцип

> Чат — одноразовий. Файли — постійні. Усе важливе з розмови має осісти у файлі ще до завершення сесії.

## Цикл роботи (для кожного агента)

### 1. Старт сесії (після /clear)

Агент читає мінімум, у такому порядку:

1. `CLAUDE.md` / `AGENTS.md` — правила (читається автоматично)
2. `10-Проєкти/<Проєкт>/<Проєкт> — Контекст.md` — лише для проєкту, над яким працюємо

**Не читати** все сховище, дашборди й чужі проєкти без потреби.

### 2. Під час сесії

- Рішення → одразу у відповідний файл (ADR, вимоги, план), а не «потім».
- Виконані пункти плану → відмічати чекбокси в `— План.md`.

### 3. Завершення сесії (перед /clear)

Агент оновлює `<Проєкт> — Контекст.md`: поточний стан, останні рішення, наступний крок. Команда користувача: **«збережи контекст»**.

## Правила токеноміки

1. **Файл-контекст ≤ 30 рядків.** Це стислий знімок стану, не журнал. Історія — у git.
2. **CLAUDE.md / AGENTS.md — короткі.** Лише правила, без довідкового матеріалу; деталі — за посиланнями, які читаються на вимогу.
3. **Точкове читання.** Спочатку `— Контекст.md`, інші файли — лише коли потрібні для задачі.
4. **Великі задачі — порціями.** Один файл/модуль за раз; між порціями стан зберігається у файли, тож /clear безпечний у будь-який момент.
5. **Без дублювання.** Один факт живе в одному файлі, решта — посилання `[[...]]`.

## Структура файлу контексту

```markdown
# <Проєкт> — Контекст
**Оновлено:** дата
**Стан:** на якому етапі плану
**Останні рішення:** 2–3 пункти
**Наступний крок:** один конкретний пункт
**Відкриті питання:** що блокує
```
